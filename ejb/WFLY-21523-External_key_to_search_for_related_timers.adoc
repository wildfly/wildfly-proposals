= Support for External Identifiers in Persistent EJB Timers
:author:            Tomasz Adamski
:email:             tadamski@redhat.com
:toc:               left
:icons:             font
:idprefix:
:idseparator:       -

== Overview

Certain users (such as those utilizing Process Automation Managers like PAM) require the ability to query EJB timers based on custom metadata associated with the timer. While it is currently possible to rely on the `Serializable` info object provided to the `TimerConfig` during timer creation, this querying method is highly inefficient. The info object must be deserialized, forcing the system to search through timers sequentially (one by one).

This proposal introduces an indexed `external_id` column to the underlying database persistence schema and exposes an extended Java API to configure and query this identifier. This enhancement allows applications to tag timers with a specific business key upon creation, thereby enabling low-latency SQL lookups directly through the EJB Timer Service.

=== User Stories

* As a developer of a long-running process engine, I want to associate a specific EJB Timer with an external ID via a standard API extension, allowing me to efficiently query the timer without scanning the entire database.

== Requirements

Implementing this feature requires modifications across the database schema, the internal SPI, and the public-facing EJB APIs.

1.  **Schema Enhancement:** Add a nullable `VARCHAR(255)` column named `EXTERNAL_ID` to the `jboss_ejb_timer` table across all supported database dialects (including PostgreSQL, H2, Oracle, DB2, MSSQL, MySQL, MariaDB, and Sybase).
2.  **Indexing:** Include a `CREATE INDEX` statement for the `EXTERNAL_ID` column in the default timer table creation SQL scripts.
3.  **API Extension (Creation):** Introduce `org.jboss.ejb3.timerservice.ExtendedTimerConfig` in the public `jboss-ejb3-ext-api` library, extending the standard `jakarta.ejb.TimerConfig`, to allow applications to pass the external ID during timer creation.
4.  **API Extension (Querying):** Introduce `org.jboss.ejb3.timerservice.ExtendedTimerService` in the public `jboss-ejb3-ext-api` library, extending `jakarta.ejb.TimerService`, providing a `getTimersByExternalId(String externalId)` method.
5.  **SPI Update:** Add a `default` method `loadActiveTimersByExternalId` to the internal `TimerPersistence` interface. Implement this efficiently in `DatabaseTimerPersistence` using an indexed `SELECT` query.

=== Non-Requirements

* This proposal does not aim to alter the Jakarta EE EJB specification itself; it acts strictly as a WildFly-specific optimization.

=== Future Work

* There is an ongoing https://github.com/jakartaee/enterprise-beans/pull/151[proposal] to incorporate this API directly into the Jakarta EE specification. If accepted, standard support can be seamlessly implemented on top of this feature.

== Backwards Compatibility

This feature is fully backward compatible. 

* **Database Schema:** The new `EXTERNAL_ID` column is nullable. The `DatabaseTimerPersistence` logic will dynamically detect if the column is missing on startup and automatically execute an `ALTER TABLE` to migrate existing tables.
* **API Compatibility:** Existing EJB applications using standard `TimerConfig` objects will seamlessly default to a `NULL` external ID. The new capabilities are completely opt-in via casting to `ExtendedTimerConfig` and `ExtendedTimerService`.

=== Default Configuration

The standard Galleon layers and default configurations will not change their underlying structure. However, the default timer table creation SQL generated by the `database-data-store` will automatically include the new column.

=== Deployments

Deployments remain unaffected unless they explicitly choose to utilize the new extended classes. The new API classes are part of the `org.jboss.ejb3` public module (provided by `jboss-ejb3-ext-api` library), which is automatically available to all EJB deployments.

== Implementation Plan

The implementation spans the public API library and several classes within the EJB3 subsystem:

1.  **API Layer:** Create `ExtendedTimerConfig` and `ExtendedTimerService` in the `jboss-ejb3-ext-api` repository under the `org.jboss.ejb3.timerservice` package.
2.  **Service Internals:** * Update `ManagedTimerService` SPI to extend `ExtendedTimerService`, and update `TimerServiceImpl` to implement the new `getTimersByExternalId` method.
    * Modify all `createTimer` methods to extract the `externalId` if the provided configuration is an instance of `ExtendedTimerConfig`.
    * Update the internal `TimerImpl` and `CalendarTimer` builders to propagate the ID.
3.  **Persistence Layer:** * Update `DatabaseTimerPersistence` SQL dialect properties to include the `EXTERNAL_ID` in `INSERT`, `UPDATE`, and `SELECT` statements.
    * Implement `loadActiveTimersByExternalId` to construct a `PreparedStatement` that explicitly filters by the new column.
4.  **In-Memory Querying:** Update `TimerServiceImpl.getTimersByExternalId` to check active in-memory timers and uncommitted transaction timers, and delegate to the database persistence layer to guarantee a complete and accurate list of timers.

== Test Plan

** Community Stability

Integration tests will be added to the WildFly basic integration test suite (`testsuite/integration/basic`).

* **Integration Test (`ExternalIdTimerTestCase`):** * Utilize an Arquillian `ServerSetupTask` to dynamically reconfigure the EJB3 subsystem to use a `database-data-store` backed by the `ExampleDS` (H2 database).
    * Deploy a Singleton EJB (`ExternalIdTimerBean`) that schedules a long-running timer using `ExtendedTimerConfig`.
    * **JDBC Verification:** Execute a raw `SELECT COUNT(*)` query within the test bean to verify that the persistence layer successfully applied the SQL schema changes and mapped the `EXTERNAL_ID` string into the database record.
    * **API Verification:** Cast the injected `TimerService` to `ExtendedTimerService` and assert that `getTimersByExternalId` successfully retrieves the timer from the database state, proving the full round-trip functionality.
    * **Negative Testing:** Assert that querying for a non-existent external ID gracefully returns an empty collection.
