=  Allow a shared single connection in the Resource Adapter

== Overview

Currently there is a 1 to 1 session with connection and session for MDB's. This feature would allow the use of a single connection, so we could have only 1 connection per MDB instead of 1 connection per MDB instance.

== Requirements


Currently the resource adapter creates a separate connection for each consumer. For example, then the activation specification for an MDB contains a maxSession attribute greater than 1, a connection is created for each MDB instance / thread.
This enhancement would allow selectable enabling of shared connections, in which consumers of the same address could share a connection between multiple sessions.


=== Hard Requirements

Create and support a new *ActivationConfigProperty* called `singleConnection` to allow for connection sharing on a consumer.

=== Nice-to-Have Requirements

=== Non-Requirements

We don't define this behaviour at the management model level. Only the configuration of the activation of an MDB is affected.

== Test Plan
                                                                                                                                                                                                                                                                            
  The singleConnection activation configuration property for Message-Driven Beans (MDBs) in WildFly, which controls whether MDB instances share a single JMS connection or create one connection per session.                                                                             

  Test Environment Setup

  - Queue: myQueue (JNDI: jms/queue/myQueue)
  - MDB Configuration: maxSession=15 for all test MDBs
  - Server Configuration: Annotation property replacement enabled in EE subsystem
  - Dependencies: Awaitility added for async verification

  ---
  Test Cases

  1. MDB Lifecycle with Single Connection (testMdbRedeploymentWithSingleConnection)

  Configuration: singleConnection=true

  Test Steps:
  1. Deploy MDB with singleConnection=true
  2. Verify exactly 1 JMS connection is created
  3. Verify 15 JMS sessions are created (matching maxSession)
  4. Send message to queue and verify MDB processes it correctly
  5. Undeploy the MDB
  6. Verify 0 connections and 0 sessions remain
  7. Redeploy the same MDB
  8. Verify exactly 1 connection and 15 sessions are recreated
  9. Send and verify message processing again

  Expected Results:
  - Single connection shared across all 15 sessions
  - Connection properly cleaned up on undeploy
  - Connection properly recreated on redeploy
  - Message processing works in all deployment states

  ---
  2. Multiple Connections Per Session (testSingleConnectionFalse)

  Configuration: singleConnection=false

  Test Steps:
  1. Deploy MDB with singleConnection=false and maxSession=15
  2. Verify exactly 15 JMS connections are created (one per session)
  3. Verify 15 JMS sessions are created
  4. Send message to queue and verify MDB processes it correctly

  Expected Results:
  - Each of the 15 sessions gets its own dedicated connection
  - Total of 15 connections created
  - Message processing works correctly

  ---
  3. Expression Support - Default Value (testExpressionSupportForSingleConnectionNotSet)

  Configuration: singleConnection="${mdb.single.connection:true}" (system property not set)

  Test Steps:
  1. Ensure system property mdb.single.connection is NOT set
  2. Deploy MDB with expression-based configuration
  3. Verify exactly 1 connection is created (default value applies)
  4. Verify 15 sessions are created
  5. Send message and verify processing

  Expected Results:
  - Expression resolves to default value true
  - Single connection behavior applied
  - Message processing works correctly

  ---
  4. Expression Support - Explicit True (testExpressionSupportForSingleConnectionTrue)

  Configuration: singleConnection="${mdb.single.connection:true}" with property=true

  Test Steps:
  1. Set system property mdb.single.connection=true
  2. Deploy MDB with expression-based configuration
  3. Verify exactly 1 connection is created
  4. Verify 15 sessions are created
  5. Send message and verify processing
  6. Undeploy and remove system property

  Expected Results:
  - Expression resolves to true from system property
  - Single connection behavior applied
  - Message processing works correctly

  ---
  5. Expression Support - Explicit False (testExpressionSupportForSingleConnectionFalse)

  Configuration: singleConnection="${mdb.single.connection:true}" with property=false

  Test Steps:
  1. Set system property mdb.single.connection=false
  2. Deploy MDB with expression-based configuration
  3. Verify exactly 15 connections are created (overrides default)
  4. Verify 15 sessions are created
  5. Send message and verify processing
  6. Undeploy and remove system property

  Expected Results:
  - Expression resolves to false from system property (overriding default)
